\chapter{Introduction}
This document details the ASW domains for the ByteRoute project, a web-oriented traffic monitoring platform. The focus is on system architecture, components, and operational choices for real-time visibility into network connection destinations using a monorepo structure.

\chapter{Project Context}
ByteRoute is a university project for the ASW (Architetture Software per il Web) course. Its aim is to deliver live network destination visibility, helping users discover the services and servers their computers connect to. The project encompasses the design and implementation of multi-component monitoring, enrichment, and visualization, limited to Linux clients, authentication for backend ingestion, and GeoIP enrichment (MaxMind GeoLite2). ASW scope centers on backend, APIs, and dashboard; the project also serves SPE coursework, but this report isolates web aspects.

\chapter{Architecture and Design}
The monorepo organizes the application into three main web-centric modules:

\begin{itemize}
    \item \textbf{Client}: A Go agent for Linux, built on libpcap. It auto-generates BPF filters based on direction and local IPs or accepts custom filters, deduplicates destination IPs, and batch-posts network connection records to the backend.
    \item \textbf{Backend}: Node.js/Express app. It authenticates incoming events (bearer JWT), resolves tenants, runs GeoIP enrichment (MaxMind GeoLite2), stores data in MongoDB, and emits real-time updates via Socket.IO. All ingestion and API endpoints require authentication, and data is isolated by tenant.
    \item \textbf{Dashboard}: Vue.js 3 SPA developed with Vite. Shows a world map, live connection flow, top ASN/country statistics, and time charts. Fetches state via REST, receives live updates via Socket.IO.
\end{itemize}

Containerization is provided through Docker and Docker Compose. The stack is routed via Traefik and typically exposes the dashboard at \url{http://localhost:8080} and backend health at \url{http://localhost:8080/health}.

Shared code and models are placed in \texttt{packages/shared}, including MongoDB schemas (users, tenants, connections) and code reused by backend and utilities.

\paragraph{Data flow}
\begin{enumerate}
    \item The client captures packets from a specific interface with a BPF filter, emits connection data.
    \item It deduplicates connections by flow or IP, aggregates to batch respecting limits, and posts to the backend with authentication.
    \item The backend authenticates, maps tenants, enriches connections via GeoLite2, inserts/upserts in MongoDB, and broadcasts through Socket.IO.
    \item The dashboard queries state via REST and displays live updates received through Socket.IO.
\end{enumerate}

\paragraph{Security and multi-tenancy}
\begin{itemize}
    \item User signup/signin returns a JWT for private API endpoints.
    \item Tenants are managed under users; connection ingestion and stats are scoped per tenant.
    \item Client-specific tokens are issued and used on the Go client for authenticated reporting.
\end{itemize}

\chapter{Implementation Notes}
The repository uses pnpm for managing dependencies and workspace orchestration. The Go client is built or run in a container, requiring root privileges for packet capture.

Relevant client flags:
\begin{itemize}
    \item \texttt{--iface}: network interface (required)
    \item \texttt{--direction}: \texttt{out} (default), \texttt{in}, \texttt{both}
    \item \texttt{--bpf}: custom BPF filter, else auto-generated
    \item \texttt{--reporter-ip}: optional WAN IP for geo-location
    \item \texttt{--dedupe}: deduplication mode (\texttt{flow} or \texttt{ip})
    \item \texttt{--max-batch-conns}, \texttt{--max-batch-bytes}: batch sizing
    \item \texttt{--idle-ttl}, \texttt{--flush}
    \item \texttt{--auth-token}: authentication for backend
\end{itemize}

Backend configuration via env variables:
\begin{itemize}
    \item \texttt{MAXMIND\_USER\_ID}
    \item \texttt{MAXMIND\_API\_KEY}
    \item \texttt{JWT\_SECRET}
    \item \texttt{MONGO\_URL}
\end{itemize}

Client environment variables:
\begin{itemize}
    \item \texttt{BYTEROUTE\_BACKEND\_URL}
    \item \texttt{BYTEROUTE\_AUTH\_TOKEN}
    \item \texttt{BYTEROUTE\_IFACE}
    \item \texttt{BYTEROUTE\_BPF}
    \item \texttt{BYTEROUTE\_REPORTER\_IP}
    \item \texttt{BYTEROUTE\_HOST\_ID}
\end{itemize}

\paragraph{API overview}
\begin{itemize}
    \item \texttt{POST /auth/signup}, \texttt{POST /auth/signin}: issues a JWT
    \item \texttt{POST /api/tenants}: creates a tenant (requires authentication)
    \item \texttt{POST /auth/client-token}: generates client token (authenticated)
    \item \texttt{POST /api/connections}: ingests connection batches (authenticated, tenant-scoped)
    \item \texttt{GET /health}: checks backend health
\end{itemize}

\paragraph{Payload}
The client sends a JSON body to \texttt{POST /api/connections} with reporter IP and an array of connection objects; the backend enriches and upserts in MongoDB.

\paragraph{CI/CD and releases}
CI is orchestrated with GitHub Actions: it builds/tests components, checks LaTeX documentation compilation, and publishes coverage. Coverage is reported for backend and client; summaries are posted via \texttt{irongut/CodeCoverageSummary}. Release processes are automated with \texttt{semantic-release}, producing GitHub releases with version bumps and changelogs.

\chapter{Validation and Results}
Verification includes unit and integration tests for backend, as well as client-backend communication and coverage checks. Deployments via Docker Compose, user/tenant registration, client token setup, and ingestion from the client demonstrated live operation. The dashboard and health endpoints were accessible as expected. Linux-only client support and reliance on external GeoIP databases are current limitations. CI provides coverage evidence via \texttt{coverage-reports} and \texttt{coverage-go}.

\chapter{Conclusion}
ByteRoute delivers modular, authenticated, multi-tenant network monitoring and visualization. The architecture supports data enrichment and realtime analytics through a modern web stack. Potential improvements are cross-platform clients, richer enrichment data, and extended analytics.
